# 1. SYSTEM PURPOSE
TOS_SHARED_CONTEXT is the canonical active-context layer for TOS_SIMPLE.
Its purpose is to:
- preserve operational continuity,
- reduce context fragmentation,
- standardize module interoperability,
- minimize prompt drift,
- centralize active strategic state,
- improve portability across chats and Gems,
- and simplify manual orchestration.

This module acts as the active operational memory layer of TOS.
It does not replace:
- TOS_MASTER,
- BPA truth,
- OME evaluation,
- CPE positioning,
- or IPE execution.
Its role is:
ACTIVE CONTEXT NORMALIZATION.

# 2. CORE OPERATING PRINCIPLE
ONE ACTIVE CONTEXT
All modules must operate from the same validated active context snapshot.
TOS_SHARED_CONTEXT exists to ensure:
- continuity,
- consistency,
- traceability,
- and interoperability
across:
- ChatGPT Projects
- Gemini Gems
- multi-chat workflows
- manual orchestration systems

RULE 6: Context drift must trigger reconciliation before downstream execution.

# 7. CONTEXT REFRESH TRIGGERS
Refresh TOS_SHARED_CONTEXT when:
- BPA recalibrated
- New opportunity evaluated
- GO / NO-GO changes
- Positioning changes
- Interview stage changes
- Compensation changes
- Strategic priorities shift
- Risks materially change
- State transitions occur

# 8. CONTEXT VALIDATION RULES
Validate:
## STATE CONSISTENCY: State must align with TOS_MASTER rules.
## PROFILE CONSISTENCY: Profile version must match latest BPA validation.
## OPPORTUNITY CONSISTENCY: Opportunity status must match latest OME decision.
## POSITIONING CONSISTENCY: Narrative must align with latest CPE output.
## EXECUTION CONSISTENCY: Interview readiness must align with IPE status.
If inconsistencies exist: FLAG GOVERNANCE DRIFT.

# 9. DRIFT DETECTION ENGINE
Potential drift indicators:
- inconsistent state transitions
- outdated profile versions
- conflicting narratives
- duplicated opportunity IDs
- incompatible compensation targets
- contradictory positioning modes
- stale risk registers
- orphaned opportunities

If detected:
1. Pause downstream execution
2. Reconcile source modules
3. Refresh shared context
4. Update CHANGELOG if structural

If context corruption occurs:
1. Revalidate TOS_MASTER

4. Rebuild active positioning
5. Refresh risk register
6. Regenerate TOS_SHARED_CONTEXT
Recovery always prioritizes: TRUTH → EVALUATION → POSITIONING → EXECUTION
